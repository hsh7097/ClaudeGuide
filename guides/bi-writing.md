# BI 작성 가이드

## 1. BI란 무엇인가

BI (Bi-Weekly) 는 팀 내에서 2주 단위로 진행하는 기술/이슈 공유를 의미한다.

BI는 발표 자료가 아니라 기술 기록이다.

목적은 설명이 아니라,
나중에 다시 봤을 때 당시 상황과 판단을 빠르게 떠올리는 것이다.

## 2. BI를 왜 저장하는가

- 작업 중 발생한 이슈의 맥락을 잊지 않기 위해
- 같은 문제를 다시 겪지 않기 위해
- "왜 이렇게 했는지"에 대한 의사결정 기록
- 팀의 기술 히스토리 축적
- 신규 인원 온보딩 시 참고 자료로 활용

즉,
BI는 개인과 팀을 위한 기술 로그다.

## 3. BI 문서의 기본 원칙

BI는 기록이다

- 기술 블로그 ❌
- 공식 문서 ❌
- 상세 가이드 ❌

→ "그때 이런 이슈가 있었고 이렇게 확인/대응했다" 정도면 충분하다.

분량은 짧고 명확하게

- 최소 5줄 이상
- 보통 5 ~ 8줄
- 한 주제당 한 화면 내에서 끝나는 분량

경험 중심으로 작성

- 이론 설명 ❌
- 내부 구조 설명 ❌
- ✔ 어떤 작업 중이었는지
- ✔ 무엇이 불편했는지
- ✔ 어떤 방법을 찾았는지
- ✔ 무엇이 가능해졌는지

## 4. BI 권장 작성 구조

### 1) 배경
- 어떤 작업을 하던 중이었는지
- 어떤 계기로 이슈를 접했는지

### 2) 문제 상황
- 기존 방식으로는 왜 확인이 어려웠는지
- 앱/로그/도구의 한계

### 3) 해결 방법 (핵심만)
- "이 방법을 사용했다" 정도만 작성
- 코드 나열은 기본적으로 제외
- 단, 나중에 다시 봤을 때 반드시 필요한 핵심 메서드라면 하나 정도 포함 가능

### 4) 결과 / 정리
- 이 방법으로 무엇이 가능해졌는지
- 왜 공유할 가치가 있었는지

## 5. BI 문체 가이드

- ❌ 공식 문서에 따르면
- ❌ 내부적으로 ~ 하기 때문에
- ❌ 왜냐하면 ~ 구조라서
- ✅ ~ 이슈가 있었다
- ✅ 이런 경우에 사용했다
- ✅ 이렇게 확인할 수 있었다

설명보다 맥락, 이론보다 경험

## 6. BI 예시

SFC 작업을 진행하면서 WebView 관련 이슈가 많이 발생하였다.

해당 이슈들은 앱 내에서만 확인이 불가능한 경우가 있어
웹 페이지 자체를 디버깅해야 하는 케이스들이 존재했다.

쉽게 디버깅할 수 있는 방법을 찾던 중
chrome://inspect/#devices 를 활용하는 방법을 확인했다.

해당 inspect를 사용하면
사용 중인 안드로이드 기기에서 실행 중인 WebView를 확인할 수 있으며,
HTML 등 WebView 내부 내용도 함께 확인이 가능하다.
