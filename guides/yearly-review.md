## 본인평가 작성방법 - 2 Questions

### What (기여도와 영향력)

> 올해 본인의 "기여도"와 "영향력"은 무엇이었나요?
> 목표했던 바 대비 결과를 어떻게 달성하였는지 설명해주십시오.

- 올해 목표는 얼마나 달성 하였나요?
- 변화하는 환경속에서 어떻게 적응하고 결과를 도출할 수 있었나요?
- 팀이나 조직의 목표달성에 기여 하였나요? 무엇을, 어떻게 기여 하였나요?
- 본인의 역할에서 '성공'은 어떤 모습이며, 올해 성과와 비교하였을 때 어떻게 생각하나요?
- 가장 자랑스러운 부분은 무엇인가요?
- 팀에서 본인이 영향을 주거나, 주도하거나, 촉진하여서 괄목할 만한 변화를 만들어낸 부분이 있나요?
- 당신의 노력이 아니었다면 이루어질 수 없었던 주요 성공사례가 있었나요?

### How (G마켓 DNA & Behaviors)

> 어떻게 G마켓 DNA를 실천하였나요?
> - 본인이 중점적으로 실천한 G마켓 DNA 중 한 두가지 예를 공유해주세요
> - 앞으로 본인의 기여도와 영향력 향상을 위해 집중하고 싶은 우리의 신념 중 한 두가지는 무엇인가요?

- 고객을 모든 행동의 중심에 두고 행동 하였나요?
- 어떻게 어려움을 극복하여 목표를 달성하거나 실패를 통해 배웠나요?
- 성과를 만드는데 있어 어떻게 올바른 일에 집중하고 있음을 확신하나요?
- 본인과 다른 견해와 의견을 언제 어떻게 구하였나요?
- 다른 이들의 업무 처리가 더 쉽도록 하거나 팀의 목표를 성취하기 위해 어떻게 하였나요?
- 사일로를 극복하고 여러 부서 간의 협업을 가능하게 하였나요?
- 다른 이를 대하거나 전반적 업무 태도에서 윤리적으로 행동하였나요?

## 본인평가 작성 Tips

1. **목표와 실제적인 성과를 함께 고려 합니다.**
   - 올해의 목표를 먼저 돌아본 후 목표의 완료여부에 대한 확인 및 그에 따른 성과를 검토합니다.
   - 조직의 변화로 인하여 목표의 변화가 많거나 목표 수립이 어려웠던 경우, 평가 기간 내 실질적인 성과, 산출물 등을 확인하여 한 해 동안의 놓치는 성과가 없도록 확인합니다.

2. **구체적으로 적습니다.**
   - 주요 기여와 영향에 대해 간결하고 명확하게 Action과 숫자를 들어 설명합니다. (불렛 포인트 방식 추천!)

3. **실제 사례를 적습니다.**
   - Challenge, 성공 case는 실제 사례를 통해 적으면 매니저가 이해하는데 도움이 됩니다.

4. **영향력에 집중합니다.**
   - "무엇 무엇을 했다"가 아닌 "무엇을 해서 어떤 영향이 있었다"를 설득해야 합니다.

## 본인평가 Good Practice

### 예시 1: 목표별 What/How 분리형

```
[ 목표 1 : 본부 GMV 목표 달성]

< What >
GMV OO 원 (달성률 XX %)

< How >
- 본부 협의체 리드를 통해 각 담당별 상황을 파악하여 문제해결을 위한 해결 방안 수립
- 본부 재무목표 달성을 위한 각 부서별 적재적소에 활용 될 수 있도록 과거 실적분석과 향후 계획을 바탕으로 Planning 진행
- 채널/조직/카테고리별 실적 분석 및 Tracking을 통해 Forecast 진행 (Daily, Weekly)
- 유관부서와 정기 커뮤니케이션 Set-Up 및 정례화를 통한 협업 증진
  (관련된 Stakeholder 의 동료 피드백 참고 요망)
```

### 예시 2: What/How 통합 서술형

```
1. What
   목표 1) 협업 프로젝트 OO 연내 런칭 : Completed
   목표 2) 멤버십 가입자 증대
      i. Target : OOO 만
      ii. Actual : OOO 만 (달성률 OO%)
   목표 3) 프로모션 예산효율화
      i. Target : OOO 만
      ii. Actual : OOO 만 (달성률 OO%)
      iii. 회사차원의 비용절감 기조에 맞추어 효율성 최대화 기조유지

2. How
   A. Work As One Team
   올해 업무를 하면서 가장 중점을 두었던 부분은 OO 프로젝트를 위한 그룹사와의 협업입니다.
   두 회사의 일하는 문화가 서로 달라 어려움이 있었고 커뮤니케이션의 방식과 의사결정 방식에 대한
   서로 간의 합의를 하는 데 많은 리소스가 투입되었습니다.
   이 과정에서 중간조율자 역할을 수행하였습니다. 이해관계자 간에 이해의 차이가 발생하는 부분에
   대해서 빠른 해결을 위해서 Weekly Meeting 을 주도적으로 셋업하였고 데이터의 원활한 공유와
   기준의 통일을 위한 Workflow 를 설계하여 그룹사와 공통으로 활용하였습니다.
```

### Good Practice 포인트 정리

- **What**: 목표별로 Target vs Actual 수치를 명시하고, 달성률(%)을 표기
- **How**: 구체적인 DNA 항목을 명시하고, 실제 사례(상황-행동-결과)로 서술
- 불렛 포인트로 간결하게 정리하되, 영향력/임팩트 중심으로 작성
- 동료 피드백 참고를 언급하면 근거가 강화됨

## 동료 피드백

### 왜 필요한가?

1. 성과평가는 평가 자체보다 나의 발전을 위한 피드백을 받는 데 더 큰 의미가 있습니다.
2. 내가 알지 못했던 나의 기여도와 영향력 (What) 을 확인할 수 있습니다.
3. 협업하는 과정에서 관찰된 나의 성과를 달성하는 모습 (How) 에 대한 의견을 들을 수 있습니다.
4. 동료 피드백을 바탕으로 본인평가 내용을 좀 더 풍부하게 작성할 수 있습니다.
5. 더욱 강화하거나 개선하고 싶은 부분에 대해 리더와 대화를 통해 향후 Development area로 활용할 수 있습니다.

### 피드백 요청 질문 프레임 (Start / Continue / Stop)

| 구분 | 내용 |
|------|------|
| Continue | 잘하는 것들, 앞으로 지속했으면 하는 것들 |
| Start | 새롭게 도전 (시작) 했으면 하는 것들 |
| Stop | 아쉬운 부분, 발전/개발 했으면 하는 부분들 |

### 4A Principle

**피드백을 주는 분:**
- AIM TO ASSIST : 상대에게 도움을 주겠다는 선의를 가지고 피드백
- ACTIONABLE : 실질적인 변화를 위한 현실적인 대안을 함께 이야기

**피드백을 받는 분:**
- APPRECIATE : 피드백에 감사한 마음을 가져 주세요
- ACCEPT OR DISCARD : 모든 피드백을 반드시 수용해야 하는 것은 아닙니다

## MISSION - 지마켓의 존재이유

> 쇼핑 경험을 더욱 즐겁게, 커머스 생태계를 더욱 건강하게 만듭니다

## WAY OF WORKING - G마켓의 일하는 방식

### 01. PUT CUSTOMERS FIRST - 모든 일의 시작과 끝에는 고객이 있어야 한다
### 02. COMMIT TO CHALLENGE - 우리가 두려운 것은 실패가 아닌, 도전을 막는 레거시이다
### 03. BE AGILE - 빠르게 시작하고, 실행하면서 완벽해지자
### 04. BE THE BRAND - 나의 일이 곧 나의 브랜드가 된다
### 05. RESPECT EACH OTHER - 토론은 격없게, 존중은 격있게
### 06. WORK AS ONE TEAM - 네 일 내 일 선 긋는 사람에게 내일은 없다
### 07. BE TRANSPARENT - 일(1)이 되려면 공유가 먼저다
### 08. ACT WITH INTEGRITY - 쉬운 방법보다 옳은 방법을 선택하자

## 목표 구조 (개발자용)

| 목표 | 설명 | 예시 마일스톤 |
|------|------|--------------|
| 목표 1 | 플랫폼 현대화 (Compose 전환 등) | 상품카드 Compose 전환 완료 |
| 목표 2 | 개발 문화/환경 개선 | 코드 리뷰 참여, 기술 문서 작성 |
| 목표 3 | 사이트 가용성 (Crash-Free Rate) | 앱 안정성 개선 |
| 목표 4 | 시스템 개선 및 운영 효율화 | 공통 모듈 개선, 성능 최적화 |
| 목표 5 | 회사 전략 프로젝트 기여 | BC 업무, 다크패턴 대응, 접근성 |

## 마일스톤 형식

```
[작업명] - [상세 설명]
예시: 상품카드 Compose 전환 1차 완료 - View + Compose 하이브리드 구조 적용
```

## 근거자료 수집 방법

| 항목 | 수집 방법 | 코멘트 예시 |
|------|----------|------------|
| GitHub PR | `is:pr author:{username} created:>={year}-01-01` | PR {N}건 - 본인 작업 |
| Code Review | `is:pr reviewed-by:{username} created:>={year}-01-01` | 코드 리뷰 {N}건 참여 |
| Confluence | `created >= {year}-01-01 AND type = page` | 기술 문서 {N}건 작성 |
| Jira | Jira에서 CSV 내보내기 | 이슈 처리 내역 |
| Firebase | Crashlytics 대시보드 스크린샷 | Crash-Free Rate (팀 성과) |

## 데이터 수집 명령어

**Git 커밋 추출**
```bash
git log --author="{username}" --since="{year}-01-01" --oneline
git log --author="{username}" --since="{year}-01-01" --stat
git log --author="{username}" --since="{year}-01-01" -- "**/*Compose*.kt"
```

**GitHub 검색 필터**
```
is:pr author:{username} created:>={year}-01-01
is:pr reviewed-by:{username} created:>={year}-01-01
is:pr author:{username} created:>={year}-01-01 repo:{org}/{repo}
```

**Confluence CQL**
```
created >= {year}-01-01 AND type = page AND creator = currentUser()
```

## 작성 시 주의사항

**피해야 할 표현**
- "버그 N건 수정" → 본인 과실로 오해 가능
- "크래시 수정" → "앱 안정성 개선 - 크래시 유발 요소 사전 제거"로 변경

**목표 분류 기준**
- 크래시 수정 → 목표 3 (사이트 가용성)
- UX 개선 (백키 시나리오 등) → 목표 4 (시스템 개선)
- 법규 대응 (다크패턴, 접근성) → 목표 5 (전략 프로젝트)
- Compose 전환 → 목표 1 (플랫폼 현대화)
- 공통 모듈 개선 → 목표 4 (운영 효율화)

**팀 성과 vs 개인 성과**
- Firebase Crash-Free Rate → 팀 성과 (스크린샷으로 첨부)
- GitHub PR 수 → 개인 성과 (본인 작업 강조)
- 코드 리뷰 → 개인 기여 (팀 협업 측면)
